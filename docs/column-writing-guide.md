# Column 執筆ガイド

Affectosphere Group の Column 記事を書くときの方針集。
新しい記事を起こすたびに、まずここに目を通す。

---

## 0. なぜラフな文体に寄せるか

初期 14 本は「研究 → リスク管理 → 価値創出 → アクション」の硬い構成で、
学術紀要の延長線上のような文体だった。これは「正確だが読まれない」リスクが高い。

note やビジネス向けの読まれる記事(けんすうさん「アル開発室」や DOWM さん等)
を参照すると、以下のような共通点がある:

- 一文が短い、改行が多い
- 「みたいな」「っぽい」「ですよね」「と思います」など口語表現
- 「最近こう思い始めた」「ちょっと考えてみました」のような書き出し
- 結論を先に出す
- 番号付きの小見出し(①②③ など)で骨格を見せる
- 自分の体験・引用元 URL・固有名詞・数字を素直に出す
- 「では!」のような柔らかい締め

このガイドは、その**ラフな読み口**を取り入れつつ、研究室発信としての
正確さは落とさないことを目指す。

---

## 1. 構成テンプレート(刷新版)

旧構成(リスク 3 + 価値 3 + アクション 5 の均整型)はやめ、
**ストーリー型**に寄せる。1 本の記事の典型骨格:

```
(1) 書き出し ── 200-400 字
    最近気付いた / 業務で見かけた問題提起。
    「最近、〇〇ってどうなんだろうって思って」みたいな入り口。

(2) この記事の結論 ── 100-300 字
    けんすうさんスタイル。先に「結論 3 点」を箇条書き or 改行で見せる。

(3) なぜそう思うか ── 1500-2000 字
    ①②③ 等の番号付きセクションで根拠を展開。
    各セクションは見出し疑問文 or 強い主張系で揃える。
    例:「①: AI 企業はすでに『現場に降りる』モードに入っている」

(4) ビジネス示唆(ライト版) ── 400-800 字
    旧構成の「リスク管理 / 価値創出 / アクション」を全部書くのではなく、
    記事に応じて 1-2 個に絞る。チェックリスト型は重く感じるので、
    本文中に「今やるなら 〇〇 が現実解」のように散らして書く。

(5) 締め ── 100-300 字
    「と思った話でした」「では!」「ちょっと厳しめの予感です」のような柔軟な結び。

(6) 参考論文 ── 必須
(7) AI 注記 ── 必須
```

字数は依然として **3000-3500 字程度**を狙う。「5 分で読める」は維持。

---

## 2. 文体ルール

### 推奨

- ですます調を基本に、たまに常体を混ぜて呼吸を作る
- 一文を短く。50-60 字以内を意識
- 段落を 2-4 文程度に短くして改行を多用
- 「みたいな」「っぽい」「ですよね」「と思います」など口語を許可
- 「とは」「とはいえ」「というわけで」「ちなみに」など転換語を活用
- 数字・固有名詞・引用元 URL は積極的に本文に出す
- 自分の体験談・気づき・違和感を 1 人称で書いてよい
- 見出しは疑問文や強い主張(例:「LLM は名前から国籍を見抜けるか?」)

### 禁止 / 抑制

- 太字記法 `**xxx**` を**使わない**(クリックスタジオで気を散らさないため)
- 学術的な「である」一辺倒の文体
- 1 段落が 5 文以上になるような重たい塊
- 「である」「すべからく」「以下に示す」など論文調表現
- 「リスク 1: ... / リスク 2: ... / リスク 3: ...」の機械的な並列を必須にしない
  (記事に応じて柔軟に)

### 例外的に許可

- イタリック `*論文タイトル*` は参考論文セクションで OK
- HTML タグ `<p class="ai-notice"><small>…</small></p>` は AI 注記でのみ OK

---

## 3. 参考論文セクション(必須・既存ルール)

各記事の本文末に必ず置く。著者表記は論文の元言語に従う。

```markdown
## 参考論文

1. Keito Inoshita (2026). *英語論文タイトル*. arXiv preprint.
2. 井下敬翔 (2025). *日本語論文タイトル*. 日本語出版先.
```

ルール:

- 英語論文 → 著者英語表記、タイトル英語、出版先英語
- 日本語論文 → 著者日本語表記、タイトル日本語、出版先日本語
- イタリックは論文タイトルのみ `*...*`
- 形式: `著者 (年). *タイトル*. 出版先, pp. ページ.`
- 1 記事が複数論文ベースなら全部列挙

frontmatter の `basedOnPaper` は主要 1 本のみ書く。

---

## 4. AI 注記(必須・既存ルール)

各記事の末尾に必ず置く。スタイルは控えめ(`.ai-notice` クラスで小さく表示される)。

日本語版:
```html
<p class="ai-notice"><small>※ 本記事は一部 AI により執筆されており、間違った情報が含まれる恐れがあります。</small></p>
```

英語版:
```html
<p class="ai-notice"><small>* This article was written in part with AI assistance and may contain inaccuracies.</small></p>
```

---

## 5. ハルシネーション回避

- 具体的な数値・モデル名・実験結果を**発明しない**
- 「論文では…と示された」「研究によれば…が確認された」程度の言い回しで包む
- 確証がない箇所は「と思っています」「予感があります」と推測であることを明示する
- 引用が必要なら必ず参考論文セクションに残す

## 6. frontmatter テンプレ

```yaml
---
title: "..."  # 見出し疑問文 or 強い主張
date: "YYYY-MM-DD"
excerpt: "..."  # 1-2 文、5 分読了の魅力を伝える
image: "/column-images/<slug>.svg"
imageAlt: "..."
readingMinutes: 5
basedOnPaper: "主要論文タイトル(原言語のまま)"
basedOnPaperUrl: "https://arxiv.org/"  # 正式 URL が判明したら差し替え
---
```

## 7. note 投稿用ファイル(必須セット)

新規記事を書いたら、必ず以下を同じ slug で揃える:

| ファイル | 内容 |
|---------|------|
| `src/content/column-ja/<slug>.md` | サイト本体・日本語 |
| `src/content/column-en/<slug>.md` | サイト本体・英語 |
| `public/column-images/<slug>.svg` | サイト本体・ヘッダ画像 |
| `src/note-export/<slug>.md` | note 投稿用整形済み本文 |
| `src/note-export/<slug>.svg` | note 投稿用画像コピー(public/column-images/ と同じ) |

note 投稿用ファイルは整形済みなので、note の入力欄にコピペするだけで投稿できる。
詳細は `src/note-export/README.md`。

## 8. 過去 14 記事のリライトについて

既存 14 記事は旧構成(リスク 3 + 価値 3 + アクション 5)で書いてある。
この新ガイドに沿って全リライトすると工数が大きいので、運用ルールとして:

- 新規記事は新ガイドで書く
- 既存記事は note 投稿時にラフな書き出しのリード文を追記する程度で OK
- 全リライトは反応の良い記事から個別に依頼してもらう

## 9. 参考にした書き手

- けんすう(アル開発室)── 一人称の語り口、疑問文見出し、結論先出し
- DOWM ── 改行多用の短文、数字強調、問いと答えの構造

両者とも、明日から実務で使える視点を、自分の体験ベースで誠実に書いている。
「研究の引用 + 個人の気付き + ビジネス示唆」の三角形が成立すると、
読者は「これは AI 生成のテンプレ記事ではない」と感じる。
